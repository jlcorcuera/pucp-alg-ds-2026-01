# Chapter 1: Asymptotic Analysis and Running Time Calculation (Iterations only)

This document serves as a technical foundation for understanding algorithm efficiency, mathematical growth rate notations, and the standardized methodologies used to calculate computational complexity.

## 1. Foundations of Algorithm Efficiency

### Algorithm Correctness vs. Efficiency

The fundamental prerequisite of any algorithm is correctness. Using the sorting problem as a baseline, consider the input array `[4, 1, 5, 6, 3]`. A correct algorithm must yield the sequence `[1, 3, 4, 5, 6]`. An algorithm that returns `[1, 6, 4, 5, 3]` is incorrect and thus fundamentally unusable, regardless of its speed.

Once correctness is verified, the design focus shifts to efficiency. Efficiency is measured by resource consumption. Comparing two correct algorithms:

* **Algorithm 1**: Requires 100 operations and 1 MB of memory.
* **Algorithm 2**: Requires 1000 operations and 10 MB of memory.

In this scenario, Algorithm 1 is the superior choice because it achieves the same result while minimizing resource utilization.

### Resource Categories

In the analysis of algorithm performance, we evaluate two primary dimensions:

* **Time Efficiency**: The speed at which an algorithm executes and returns a result.
* **Space Efficiency**: The amount of memory required during execution, calculated as the total sum of the space occupied by all variables within the algorithm.

### Hardware Independence

Technical analysis must transcend the specific hardware used for execution. Consider an experiment running the same sorting algorithm on two different machines:

1. **Laptop**: Intel Core i7, 12 GB RAM.
2. **Server**: Intel Core i9, 128 GB RAM.

While the Server executes the task faster in absolute seconds, experimental data shows that the growth behavior—the curvature of the execution time as input size increases—is identical for both. Therefore, we measure complexity as a mathematical function of input size `(n)`, ensuring the analysis remains hardware-independent.

## 2. Execution Time Measurement and T(n)

### Mathematical Representation

The execution time `T(n)` for an input of size `n` is modeled by the following relation: 
`T(n) ≈ c_op * C(n)`
Where `c_op` is the execution time of a basic operation on a specific machine and `C(n)` is the number of times that operation is executed.

### Effect of Input Size

The core of complexity analysis is determining how `T(n)` scales as `n` grows. For example, if an operation count is defined as `C(n) = 1/2 * n(n-1) ≈ 1/2 * n^2`, and we double the input size to `2n`, the impact is: 
`T(2n) / T(n) ≈ (1/2 * (2n)^2) / (1/2 * n^2) = 4`
In this quadratic growth scenario, doubling the input size results in a fourfold increase in execution time. Note that constant factors like `c_op` and `1/2` become irrelevant in this ratio.

### Execution Scenarios

To establish a reliable metric for `T(n)`, we categorize execution based on input variety:

* **Best Case**: Execution under optimal conditions; generally discarded as too optimistic for engineering guarantees.
* **Worst Case**: The maximum execution time for any input of size `n`. This is the standard definition of `T(n)` used to provide a guaranteed temporal upper bound.
* **Average Case**: The expected execution time; often mathematically difficult to characterize in practice.

### Factors Influencing Time

Execution time is the product of four distinct factors:

1. The specific input data provided.
2. The quality of the code generated by the compiler.
3. The nature and speed of the machine’s instruction set.
4. The inherent temporal complexity of the base algorithm.

## 3. Asymptotic Analysis and Notations

### Concept of Asymptotics

Asymptotic analysis evaluates algorithm behavior as the input size `n` approaches infinity (`n → ∞`). This approach prioritizes growth rates over constant factors or performance on small datasets, which are usually negligible in large-scale systems.

### Big O Notation (O)

Big O defines an asymptotic upper bound. 
**Definition**: `T(n) = O(f(n))` if there exist a positive constant `c` and a non-negative integer `n_0` such that `T(n) ≤ c * f(n)` for all `n ≥ n_0`.

Consider `f(n) = 1000n` versus `g(n) = n^2`. Although `1000n` is larger for small `n`, `n^2` eventually dominates. The crossover point is exactly `n = 1000`. Beyond this point, the quadratic function grows faster; thus, `1000n ∈ O(n^2)`.

### Omega (Ω) and Theta (Θ) Notations

* **Omega (Ω)**: Represents a lower bound (`f(n) ≥ c * g(n)`).
* **Theta (Θ)**: Represents a tight bound (`c_1 * g(n) ≤ f(n) ≤ c_2 * g(n)`), indicating that `f(n)` and `g(n)` grow at the same rate. This is the most precise notation for describing complexity.

### Notational Insights and Validity

A key insight in Big O notation is that the base of a logarithm is irrelevant. Since `log_a(n) = log_a(b) * log_b(n)`, and `log_a(b)` is a constant, there is no effective difference between logarithmic bases in asymptotic terms.

**Examples of Asymptotic Claims:**

* `n ∈ O(n^2)`: True
* `100n + 5 ∈ O(n^2)`: True
* `n^2 ∈ O(n^3)`: True
* `n^3 ∈ O(n^2)`: False

**Logic for Proving `n^2 ∈ O(n^3)`:**

1. Assume `n^2 ≤ c * n^3` for `n ≥ n_0`.
2. Simplify: `n^2(1 - c * n) ≤ 0`, which implies `c * n ≥ 1`.
3. Solve for `n`: `n ≥ 1/c`. By selecting `c = 2` and `n_0 = 1`, the condition is satisfied.

## 4. Common Growth Rates and Comparative Analysis

### Growth Rate Hierarchy

Algorithms are classified by their characteristic growth rates, ranked here from most to least efficient:

| Function | Name | Example |
|---|---|---|
| `c` | Constant | Adding two numbers |
| `log n` | Logarithmic | Balanced binary tree search |
| `n` | Linear | Searching an unsorted list |
| `n log n` | Linearithmic | Mergesort or Quicksort |
| `n^2` | Quadratic | Bubble Sort or Insertion Sort |
| `n^3` | Cubic | Matrix multiplication (`n × n × n`) |
| `2^n` | Exponential | Brute-force solutions |

### Growth Acceleration

As `n` scales, the disparity between these functions becomes extreme:

| n | n ln n | n^2 | n^3 | 2^n |
|---|---|---|---|---|
| 1 | 0 | 1 | 1 | 2 |
| 16 | 64 | 256 | 4,096 | 65,536 |
| 256 | 2,048 | 65,536 | 16,777,216 | `1.16 × 10^{77}` |
| 65,536 | 1,048,576 | `4.295 × 10^9` | `2.815 × 10^{14}` | Immense |

## 5. General Rules for Calculating Complexity

### Asymptotic Simplification Rules

* **Rule 1 (Sums/Products)**: For additions, the maximum order dominates (`T_1 + T_2 = max(O(f), O(g))`). For products, the orders are multiplied (`T_1 * T_2 = O(f * g)`).
* **Rule 2 (Polynomials)**: If `T(x)` is a polynomial of degree `n`, then `T(x) = Θ(x^n)`.
* **Rule 3 (Logarithms)**: `log^k n = O(n)` for any constant `k`.
* **Constant Factor Rule**: `O(k * f(n)) = O(f(n))`. Multiplicative constants and lower-order terms (e.g., the `n` in `n^2 + n`) are ignored.

### Instruction-Based Rules

* **Sequential Instructions**: Calculated by summing the OE of each individual instruction.
* **Loops (Rule 1)**: The time is the number of iterations multiplied by the operations within. The formula for the worst case is: `1 (init assign) + 1 (extra comp) + Σ (body + 2 (inc/assign) + 1 (comp))`
* **Nested Loops (Rule 2)**: Analyzed "inside-out." Total time is the product of the sizes of all loops. A nested structure with `x` instructions in the inner body follows: `2 + 5n + (3+x)n^2 ∈ O(n^2)`.
* **Function Calls (Rule 4)**: `T = 1 + Σ T(P_i) + T(F)`, where `1` is the call cost, `T(P_i)` is the time to evaluate the `i`-th parameter, and `T(F)` is the function's internal execution time.
* **Conditionals (Rule 5 & 6)**:
  * `if`: Cost of condition evaluation + cost of the body.
  * `if-else`: Cost of condition + `max(cost of 'then' branch, cost of 'else' branch)`.

## 6. Code Complexity Examples

### Example 1: Single Loop Analysis

Consider a loop performing array multiplication and summation:

1. **Body OE Breakdown** (9 total):
  * `c[i] = a[i]*b[i]`: 5 OE (3 accesses, 1 multiplication, 1 assignment).
  * `suma = c[i] + suma`: 3 OE (1 access, 1 addition, 1 assignment).
  * `cout`: 1 OE.
2. **Derivation**: `1 + 1 + Σ(9 + 2 + 1) = 2 + 12n ∈ O(n)`.

### Example 2: Nested Loops (Bubble Sort)

Applying Rule 2 to the "Burbuja" algorithm using specific OE counts from the internal logic:

* **Inner Body** (13 OE): An `if` check (4 OE: 2 accesses, 1 comparison, 1 subtraction) plus the swap body (3 OE + 4 OE + 2 OE = 9 OE).
* **Inner Loop Overhead**: 5 OE (init, comparison, subtraction, increment/assignment).
* **Total Instructions**: `1 + 2 + Σ(#for_interno + 2 + 2)`. This nested structure results in `O(n^2)`.

### Example 3: Function Subroutine (Intercambiar)

The function `intercambiar` accepts two parameters and performs 3 internal operations. Applying Rule 4: `T(n) = 1 (call) + 2 (parameter evaluation) + 3 (body) = 6`. Since the result is a constant, the complexity is `O(1)`.

### Example 4: Summation (Sumatoria)

Calculating `T(n)` for a standard summation algorithm (`s=0; for... s+=a[i]`):

* `s=0`: 1 OE.
* **Loop initialization and extra comparison**: `1 + 1 = 2` OE.
* **Loop Body** (`s += a[i]`): 3 OE (access, addition, assignment).
* **Loop Control** (`i++, i<n`): `2 + 1 = 3` OE.
* **Total OE**: `1 + 2 + Σ(3 + 3) = 3 + 6n ∈ O(n)`.
